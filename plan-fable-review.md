Reviewed: all `.rs` sources in `crates/{rmc-core, rmc-stats, rmc-grids, rmc-numeric, rmc-io, rmc-frohlich, rmc-minimal, rmc}`, the criterion bench `rmc-core/benches/hot_path.rs`, the `rmc-minimal` throughput harness, and the Python drivers in `bench/`.

Implementation status checked against the current tree:

- Done: A1, U1, U3, A2, Q1, A4, Q6.
- Partial: U2. Core now has `RunPlan`/`run_plan_full` with warmup, progress callbacks, parallel merge, final kernels, and final states, and `rmc-frohlich` uses it. The generic results-directory writer proposed in the review is still app-local.
- Not seen in the current tree: other table items beyond U1/Q6.

## 1. Performance

**Summary.** The hot path is fundamentally sound and shows real care. The static, monomorphized update-set path (`SingleUpdateSet`, `TwoUpdateSet`, `WeightedUpdateSet`) keeps RNG draws and update dispatch fully inlined; heterogeneous updates are handled by a user-written enum (`PolaronUpdate`, `MinimalUpdate`) instead of `Box<dyn Update>`, so the per-step loop in `step_update` (sets.rs:855) does zero allocation and zero virtual dispatch. The earlier `O(num_bins)`-per-sample measurement bug was fixed properly (`BinnedBatchedSums`, measurement.rs:95) *and* locked in with an equivalence test. `Diagram` uses a slotmap arena plus an index-tracked `storage` vec, so random vertex selection is O(1) and insert/remove are O(segment) as the physics requires. Sanity checks are `debug_assert!`-gated and fully compile out in release. Parallelism is deterministic (per-chain seeded xoshiro streams, order-preserving fold). Nothing here undermines the measured ~1.8× advantage over the C++ reference. The remaining findings are second-order.

**Issues.**

- **[P1, MEDIUM]** `ScalarAutocorrelation::accumulate` is O(max_lag) *twice* per sample — `rmc-stats/src/lib.rs:643-651` (lag loop, inherent) plus `rmc-stats/src/lib.rs:659` `self.history.remove(0)`, which memmoves the whole history buffer on every sample once warm.
  **Fix:** replace the `Vec<f64>` history with a fixed ring buffer: keep `history: Vec<f64>` at capacity `max_lag`, add a `head: usize` cursor, index `previous` as `history[(head + max_lag - lag) % max_lag]`, and overwrite `history[head]` then `head = (head + 1) % max_lag`. This drops the memmove entirely and keeps merge semantics (clear = reset `len`/`head`). ~10 lines, no API change.

- **[P2, MEDIUM]** The criterion bench does not cover the production hot path. `rmc-core/benches/hot_path.rs:69-117` benchmarks only `SingleUpdateSet` (one update, always-accept) and the dyn boxed set — but *both* real consumers (`rmc-minimal/src/lib.rs:167`, `rmc-frohlich/src/updates/mod.rs:74`) run `WeightedUpdateSet` with 3–8 enum variants. The costs that actually dominate a real step — `WeightedIndex::sample` (binary search + `gen_range`) at sets.rs:696, enum-match dispatch, and a per-cycle measurement accumulation — are unmeasured in criterion. The gap is partially compensated by the end-to-end `rmc-frohlich bench` subcommand and `rmc-minimal`, but those cannot attribute regressions to a specific layer.
  **Fix:** add two benches: (a) `WeightedUpdateSet<enum{8 variants}>` with a realistic accept/reject mix, and (b) the same run with a `ScalarBlockMeans`-style measurement at `steps_per_cycle = 5` (the production cadence). That makes the criterion suite able to catch regressions in the code paths the physics actually uses.

- **[P3, LOW]** `IndicatifProgress::on_step` (`rmc-core/src/mc/progress.rs:47-53`) calls `bar.inc(delta)` on every MC step. indicatif throttles drawing but still does atomic position updates per call; at 10⁸–10⁹ steps this is measurable overhead in the default interactive path (`run_from_config_with_progress` is what `rmc-frohlich` `main` runs). The timed `run_bench` path avoids it, so benchmark numbers are clean.
  **Fix suggestion:** move increments to `on_cycle` (add `bar.inc(steps_per_cycle)` there) or only `inc` every N-th step.

- **[P4, LOW]** `PolaronMeasurement::measure` reconstructs the `LinearGrid` from `GridSpec` on every measurement cycle (`rmc-frohlich/src/measurement.rs:416` → `GridSpec::grid()` at measurement.rs:198, which runs `LinearGrid::new` validation + division). Trivial per call but pure waste at ~1 call / 5 steps; cache the grid in `PolaronMeasurement` (it is `Copy`-cheap: `LinearGrid` is `Copy`).

- **[P5, LOW]** `DynUpdateSet` keeps two copies of the counters — on `UpdateEntry` (entries.rs:79-84) and mirrored into `stats: Vec<UpdateStats>` with an indexed write per step (sets.rs:367-375) plus a full re-collect in `refresh_stats` (sets.rs:427-438). Not hot in production (the dyn path is unused there, see A1), but it is a needless double bookkeeping that showed up once already as the per-step `refresh_stats()` bug the comment at sets.rs:364 describes.

- **[P6, INFO]** Cache/layout of the update-set structs is fine as-is: `WeightedUpdateSet` stores entries, stats, and the `WeightedIndex` in separate contiguous vecs (SoA-ish), and `TwoUpdateSet` is a flat struct of arrays (sets.rs:713-720). `Vertex` (diagram.rs:11-20) is ~120 bytes with three `Vector3<f64>`; the linked-list walks in `get_p_mean_range`/`exact_estimator` are inherent to the algorithm (they mirror the C++), and slotmap keeps them in one contiguous slab. No action needed.

- **[P7, INFO]** Rayon is used correctly where it matters (independent chains, `rmc-core/src/mc/parallel.rs:119`, `rmc-frohlich/src/app.rs:256`). Intra-chain parallelism is not applicable to Markov chains, so nothing meaningful is "left on the table" beyond the fact that the frohlich app bypasses `run_parallel` (see A2/U3) — that is an architecture problem, not a throughput one.

## 2. Usability

**Summary.** The minimum contract is genuinely small: a user implements exactly **two traits** — `Update<State>` (attempt/accept/reject) and `Measurement<State>` (measure/finish) — supplies any owned `State`, and calls `run_typed` or `run_parallel`. Results flow back by ownership with no `Any`/downcasting, `Merge` makes the parallel reduction contract explicit, and `SeedSource`/`ChainId` give reproducibility for free. `rmc-minimal/src/lib.rs` (180 lines for a full 3-update model with jackknifed observables) is a fair advertisement. Docs are unusually good — the tricky semantics (RNG stream consumption on the accept branch, partial final cycles, Merge for sums-not-means) are all called out where they bite.

**Issues.**

- **[U1, MEDIUM, DONE]** Enum-dispatch boilerplate. To mix update types in one `WeightedUpdateSet`, the user must hand-write a wrapper enum plus three exhaustive match blocks: `rmc-minimal/src/lib.rs:27-57` (31 lines for 3 variants) and `rmc-frohlich/src/updates/mod.rs:46-72` (27 lines for 8 variants, and a fourth match for `name()`). This is pure mechanical noise that every new model pays, and it grows linearly with variant count × trait-method count.
  **Current status:** implemented as `rmc_core::dispatch_update!`, which generates the enum, `Update<State>` dispatch, default or forwarded `reject`, and a default `name()` from the variant identifier. `rmc-minimal` and `rmc-frohlich` now use it.
  **Fix:** ship a `macro_rules!` (or derive macro) in rmc-core, e.g. `dispatch_update! { pub enum MinimalUpdate<MinimalState> { Gaussian(GaussianShift), Uniform(UniformShift), Mirror(Mirror) } }` that expands to the `Update<State>` impl (and optionally a `name()`). This removes ~30 lines per model and eliminates the forgot-a-match-arm-in-`reject` failure mode.

- **[U2, MEDIUM, PARTIAL]** The framework absorbs the run loop but not the *application harness*. Everything in `rmc-frohlich/src/app.rs` (547 lines) other than physics — warmup phase, per-chain progress bars in a `MultiProgress`, parallel chain orchestration + output merge, checkpoint payload, results-directory writing — would need to be re-written nearly verbatim for a second model. `rmc-minimal/src/main.rs` (217 lines) re-implements a smaller version of the same (warmup, rep timing, summary printing). Concretely, a second physics model costs: state + updates + measurement (cheap, fine) **plus ~300–500 lines of copied harness** (expensive, avoidable).
  **Current status:** partially implemented. `rmc-core::mc::RunPlan` plus `run_plan_full`/`run_plan_full_with_callbacks` now cover warmup, callbacks/progress, parallel merge, final kernels, and final states. `rmc-frohlich` uses that path. Result writing/checkpoint manifest code remains app-local.
  **Fix:** extract a driver layer — either into rmc-core or a new `rmc-app` crate — providing: `RunPlan { warmup: SimulationParams, main: SimulationParams, chains, seed }`, a `run_plan(plan, build, progress: bool) -> (SimulationStats, Output, Vec<PerChainUpdateStats>)` that owns warmup + progress + merge, and a generic `write_results(dir, serializable_sections...)` manifest writer. app.rs then shrinks to physics-specific config and summary formatting.

- **[U3, MEDIUM, DONE]** `run_parallel`'s `build` closure hands ownership of the kernel to the runner and never returns it (`rmc-core/src/mc/parallel.rs:119-127` discards everything but `(stats, output)`), so per-update acceptance statistics — which live in the update set inside the kernel — are unrecoverable on the parallel path. This is precisely why `rmc-frohlich` bypasses `run_parallel` and hand-rolls chains (app.rs:253-277) with its own `merge_outputs` (app.rs:414). The core API loses to its own workaround in the reference app.
  **Current status:** implemented via `run_parallel_full`, `run_parallel_full_with_callbacks`, and the run-plan variants returning per-chain kernels.
  **Fix:** return the kernel per chain, e.g. change the internal map to keep `(stats, output, kernel)` and add `run_parallel_full(...) -> Result<(SimulationStats, M::Output, Vec<K>)>`, or accept a `finalize: Fn(K) -> F where F: Merge + Send` hook that folds kernels the same way outputs are folded. Either variant lets frohlich (and any future model) delete its parallel-orchestration code.

- **[U4, LOW]** The `attempt` protocol encodes three outcomes in an `f64` (`< 0.0` = impossible, `>= 1.0` = certain accept, else probability) — traits.rs:18-23. It is fast and documented, but nothing stops a buggy weight-ratio computation from going slightly negative and being silently counted as "impossible". At minimum add `Update::IMPOSSIBLE: f64 = -1.0` as a named constant and mention the NaN behavior (NaN fails both `< 0.0` and `>= 1.0` and the uniform compare, so NaN → rejected — worth stating).

- **[U5, LOW]** Error-type story is split three ways: `RmcError` is stringly-typed (every variant wraps `String`, error.rs:5-18), `GridError`/`NumericError`/`IoError` are nicely structured, and application code gives up and uses `Box<dyn Error>` (`rmc-frohlich/src/app.rs:23`). There are no `From` bridges between the domain errors and `RmcError`. This is livable, but a top-level `enum` with `#[from] GridError | NumericError | IoError` variants (in the `rmc` facade) would let apps use one `?`-friendly result type.

- **[U6, LOW]** Three measurement APIs coexist — typed `Measurement` (returns `Output`), `SinkMeasurement` (writes named serialized artifacts), and boxed `DynMeasurement` (side-effect only, `State = ()`). The typed one is clearly the blessed path (docs say so), but a newcomer reading `mc::mod.rs` re-exports sees all three with equal prominence. Consider module-level doc guidance ("start with `run_typed`/`run_parallel`; the sink and dyn paths are for X") — or see A1 for the stronger option.

- **[U7, LOW]** `SimulationParams::default()` has `max_steps: 1` (run.rs:20-27) — a default that runs one step is a foot-gun; requiring explicit construction (no `Default`) would be safer. Also `cycles_per_check: 0` meaning "disabled" is documented but easy to miss.

## 3. Architecture

**Summary.** The crate decomposition is right: `rmc-core` is dependency-light (no stats/grids/io), statistics/grids/numerics/io are siblings consumed à la carte, `rmc` is a feature-gated facade, and the physics (`rmc-frohlich`) sits entirely outside the framework — no physics type appears in any core signature. The central abstraction choices are good ones: `State` as an owned generic threaded by the run loop (no `Arc<Mutex>`), `Kernel`/`SteppingUpdateSet` separation leaving room for non-Metropolis kernels, and `Merge` as the single reduction contract shared by stats accumulators, run stats, and user outputs. The one collapsed `drive()` loop (run.rs:90) serving all entry points is a genuinely nice consolidation.

**Issues.**

- **[A1, MEDIUM, DONE]** The runtime/boxed ("dyn") half of rmc-core is dead weight in practice. `DynUpdateSet`, `DynMeasurementSet`, `UpdateEntry`/`MeasurementEntry`, `DynUpdate`/`DynMeasurement`, `NamedSet`, `DynMetropolisKernel`, and the `run`/`run_with_callbacks`/`run_dyn_with_sink*` entry points together account for roughly half of the `mc` module (entries.rs, wrappers.rs, named_set.rs, sets.rs:13-460, kernel.rs:18-92, run.rs:143-222), yet they are used **only** by core tests, the bench, and facade examples — neither `rmc-frohlich` nor `rmc-minimal` touches them (verified by grep). Worse, the dyn path is hard-limited to `State = ()`, so it cannot express any real model whose state is a lattice or diagram; updates would have to smuggle state through interior ownership, contradicting the framework's own stated design ("owning the mutated data in `State`... is preferred", traits.rs:9-10).
  **Current status:** implemented by deleting the dyn update/kernel path. The remaining boxed API is the separate `SinkMeasurementSet<State>` output-sink path.
  **Fix:** either (a) delete the dyn path and keep the typed path as the single API (tests that exercise the run loop can use static sets), or (b) if runtime-composed update sets are a real requirement (e.g. config-driven update menus), make the dyn path state-generic — `Box<dyn UpdateObject<State>>` with the `dyn RngCore` bridge already used in wrappers.rs works for any `State` — and drop the `State = ()` specialization. Option (b) is a ~50-line change to wrappers.rs/entries.rs and would make the dyn path actually usable; option (a) removes ~600 lines and the double-bookkeeping in P5.

- **[A2, MEDIUM, DONE]** The reference app re-implements framework concerns. `run_from_config_with_progress` (app.rs:246-277), `run_single_chain_with_id*` (app.rs:315-397), and `merge_outputs` (app.rs:414-431) duplicate `run_parallel`'s seeding/merging logic with ad-hoc error-to-`String` conversion (app.rs:260-268) because the core API cannot express warmup, per-chain progress, or kernel-stat recovery (see U3). Framework/physics separation is clean in the *type* sense but leaks in the *orchestration* sense — the framework's parallel runner is not used by the only production simulation.
  **Current status:** implemented. `rmc-frohlich::app::run_from_config_with_progress` builds a `RunPlan` and calls `run_plan_full`/`run_plan_full_with_callbacks`; the old per-chain orchestration and custom merge path are gone.
  **Fix:** same as U2/U3 — grow `run_parallel` (warmup params + kernel recovery + optional per-chain callback factory already exists) until app.rs can be expressed in terms of it, then delete the duplication.

- **[A3, LOW]** `PolaronStats` mixes three kinds of state: mergeable accumulators, self-consistency *control* state (`energy_estimate`, `self_consistent_period`, ...), and immutable physics params. Its `Merge` impl silently takes `other.energy_estimate` and `other.self_consistent_period` (measurement.rs:333, 336), so the merged value depends on chain fold order — deterministic, but arbitrary. Split the control state into a non-merged field or document the "last chain wins" rule.

- **[A4, LOW, DONE]** Reset-desync in the self-consistent reweighting: `reevaluate_energy_estimate` resets `energy` and `a` mid-run but not `zeroth` (measurement.rs:399-403), after which `jackknife_energy` pairs energy-batch means over the post-reset window against zeroth-batch means over the whole run (measurement.rs:283-292). If the zeroth-sector fraction is stationary post-warmup the bias is negligible (and validation hits the target E), but the invariant stated at measurement.rs:257-258 ("shared global sample schedule") is genuinely violated for `energy`/`a` after the first re-estimate. Worth a comment at minimum; resetting a paired `zeroth_for_energy` copy would make it exact.
  **Current status:** implemented with `zeroth_for_energy`, which is used as the denominator for energy/quasiparticle jackknife estimates and reset alongside `energy`/`a`.

- **[A5, LOW]** `rmc-core/src/scalar.rs` (`Scalar`, `SampleType`) is exported framework surface used by nothing except its own foundation test — no accumulator or grid is generic over `Scalar`. Either wire it into `rmc-stats` (complex-valued moments) or remove it until needed.

- **[A6, LOW]** Facade inconsistency: the `rmc` facade exists with feature-gated re-exports, but both in-workspace consumers depend on `rmc-core` + siblings directly. Fine for a workspace, but decide whether the facade is the public story and point examples/docs one way.

- **[A7, LOW]** `rmc-io` pins `bincode = "1"` directly instead of via `[workspace.dependencies]` (crates/rmc-io/Cargo.toml) — the only dependency in the workspace managed out-of-band.

## 4. Code Quality

**Summary.** Quality is high. There is **zero `unsafe`** anywhere in the workspace. The test suite is substantial (~200 `#[test]`s): property tests prove merge commutativity/associativity *and* merge-vs-single-pass equivalence for the stats accumulators (`rmc-stats/tests/merge_properties.rs`); the O(1) binned-measurement optimization is pinned to the old semantics by an explicit equivalence test (measurement.rs:538); Ising validation checks physics against exact finite-size results (`rmc-core/tests/ising_validation.rs`); and every frohlich `accept()` runs a full structural sanity check (`sanity.rs`) under `debug_assert!`, covering list/arena consistency, link reciprocity, momentum conservation, time ordering, and `phonons_above` counts. Naming is consistent, cloning is disciplined (the few `clone()`s are config/setup-path only).

**Issues.**

- **[Q1, HIGH, DONE]** Round-robin batch assignment biases jackknife error bars. `BatchedSum::push` and `BinnedBatchedSums::push` assign sample *i* to batch `i % n_batches` (`rmc-frohlich/src/measurement.rs:27` and `:125`). For an autocorrelated MC series this interleaving is the wrong blocking: temporally adjacent (strongly correlated) samples land in *different* batches, so all batch means share the same temporal fluctuations and are mutually positively correlated. The jackknife over batch means (measurement.rs:482) assumes approximately independent batches; positive inter-batch correlation makes the reported `stderr` an *underestimate* of the true error, potentially by a factor approaching √(2τ_int) — the batching then does nothing to absorb autocorrelation at all. The point estimates (E, Z, Σ) are unaffected; only the uncertainties are.
  **Current status:** implemented. `BatchedSum` and `BinnedBatchedSums` now use contiguous batches via `batch_len = ceil(expected_samples / n_batches)` and `batch = next / batch_len`.
  **Fix:** switch to contiguous blocking — `let batch = (self.next / batch_len) % self.n_batches` with `batch_len = ceil(expected_samples / n_batches)`, or simpler: `let batch = self.next / batch_len` with a fixed `batch_len` from config and let `n_batches` grow (cap for jackknife). All series (`zeroth`, `exact`, `hist`, `energy`, `a`, `order`) must keep sharing one schedule so the ratio pairing stays valid — that property is preserved by any deterministic function of `next`. Then validate: stderr from contiguous blocks should *grow* until batch_len ≫ τ_int and plateau; compare against the C++ reference's binning analysis. If the round-robin scheme was deliberately inherited from the C++ reference, document that and the resulting caveat prominently — but note the project memory already established the C++ reference is not authoritative.

- **[Q2, MEDIUM]** Naive variance formula with cancellation risk in two accumulators. `ScalarBatchMeans::sample_variance`/`population_variance` and `ScalarBlockMeans` compute `raw_sum_sq / n − mean²` (`rmc-stats/src/lib.rs:827-840` and `:1002-1015`). For data with large mean and small variance (mean² ≫ variance) this cancels catastrophically — and it sits in the same file as the Welford-based `ScalarMoments` that exists precisely to avoid it.
  **Fix:** embed a `ScalarMoments` for the raw stream (replacing `raw_count/raw_sum/raw_sum_sq`) and delegate `mean`/`population_variance`/`sample_variance` to it. Merge already exists on `ScalarMoments`, so `Merge` for both types simplifies too. Zero API change; the `sum()`/`sum_squares()` accessors can be reconstructed or deprecated (`sum_squares` would need keeping the raw sum-of-squares only if someone consumes it — grep shows only tests do).

- **[Q3, MEDIUM]** Fallible merges panic. `ScalarAutocorrelation::merge`, `ScalarBatchMeans::merge`, `ScalarBlockMeans::merge`, `VectorMoments::merge`, `VectorCovariance::merge` all `assert!`/`assert_eq!` on shape mismatches (`rmc-stats/src/lib.rs:668, 845, 1020, 1268, 1425`), and `update_stats::merge_pair` asserts name/length equality (`rmc-frohlich/src/update_stats.rs:79-84`). Inside `run_parallel`'s fold this converts a configuration mistake (chains built with different `n_batches`) into a panic that unwinds through rayon rather than an `Err`.
  **Fix:** since `Merge::merge(self, other) -> Self` is infallible by design, keep the trait but have the parallel `build` path make mismatches impossible (all chains are built from one closure — they are, today), and *document* on `Merge` that implementations may panic on structurally incompatible operands. Alternatively add `try_merge` and make the fold in parallel.rs use it. The documentation route is cheapest and honest.

- **[Q4, LOW]** Dead code: `Diagram::add_phonon` (`rmc-frohlich/src/diagram.rs:351-353`) is an unused one-line alias of `insert_arc`; `UpdateEntry`'s `cumulative_nprops/naccs/nimps` and `accumulate_counters`/`reset_run_counters` (entries.rs:82-134) are exercised only by tests, never by any runner; `rmc-core/src/scalar.rs` per A5. None harmful; all removable.

- **[Q5, LOW]** Panic-style validation inconsistency: `BatchedSum::new`/`BinnedBatchedSums::new` `assert!` on `n_batches >= 2` (measurement.rs:17, 108) while the framework convention is `Result` (`ScalarBatchMeans::new` returns `Err` for the same class of input, rmc-stats/src/lib.rs:724-729). Also `GridSpec::grid()` `.expect()`s (measurement.rs:199) — fine given config validation, but the config (`RunConfig`) is never validated as a whole (e.g. `num_bins = 0` would panic deep in measurement construction). A `RunConfig::validate()` called in `main` would front-load these.

- **[Q6, LOW, DONE]** `NoopMeasurement` is re-implemented three times (`rmc-minimal/src/lib.rs:157-165`, `rmc-frohlich/src/app.rs:532-541`, plus test copies). A framework-provided `struct NullMeasurement;` with `impl<S> Measurement<S> for NullMeasurement` would serve all warmup phases.
  **Current status:** implemented as `rmc_core::mc::NullMeasurement`; `rmc-minimal` and `rmc-frohlich` use it.

- **[Q7, LOW]** Minor tidiness: `MinimalState`'s manual `Default` could be derived (rmc-minimal/src/lib.rs:14-18); `SinkMeasurementSet::get_by_name_mut` uses a `match` where `.map()` reads better (sets.rs:190-194); `WeightedUpdateSet::weights()/ratios()` allocate fresh `Vec`s per call (sets.rs:663-669) — fine off the hot path, but `impl Iterator` would be cheaper; `parallel.rs` duplicates its ~15-line merge-fold in both impl functions (parallel.rs:130-139 vs 175-184) — extract a helper.

- **[Q8, INFO]** Test-coverage gaps worth closing (not defects): no test pins `DynUpdateSet::add_pair`/`rebuild_distribution` ratio semantics against `WeightedUpdateSet::inverse_pair` (the two implementations of the same detailed-balance bookkeeping); nothing tests `drive()`'s partial-final-cycle measurement behavior explicitly (it is documented at run.rs:33-37); `save_json_atomic`'s tmp-file collision behavior under concurrent writers is untested (rmc-io/src/lib.rs:193-208).
